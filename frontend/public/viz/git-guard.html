💡 설계 원칙 (v2.7.0)
LLM 없는 작업은 Bash
ESLint, npm test — 판단이 불필요한 작업은 Bash 직접 실행. sub-agent spawn 비용 없음.
1-level 중첩만 사용
메인 → Agent (P-2/P-3)만. Agent 안에서 Agent spawn 금지. 결과 잘림 없음.
스코프 유닛테스트
커밋 A-2: staged 소스 2개↑ 조건부 실행. 전체 회귀는 push P-1에서만.

4 스킬 구조 & 커스터마이징
📁 스킬 파일 구조
git-guard/
├── SKILL.md                     ← 라우터 (작업 감지 → reference 지정)
├── static-analysis.config.yml  ← 정적 분석 도구 on/off
├── git-platform.config.yml     ← 플랫폼 설정 (gitea|github|gitlab)
├── git-guard-flow.html    ← 이 문서
├── scripts/
│   ├── git-platform.js         Node.js CLI (Gitea/GitLab REST, GitHub gh)
│   └── git-platform-go/        Go 소스 참조 (빌드 시 exe 전환 가능)
└── references/
    ├── branch.md                브랜치 생성 + 코드 작업 보호
    ├── commit.md                커밋 3단계 + 병렬 처리
    ├── pr.md                    PR 라우터 (플랫폼별 위임)
    ├── pr-gitea.md              Gitea REST API 절차
    ├── pr-github.md             GitHub gh CLI 절차
    ├── pr-gitlab.md             GitLab REST API 절차
    ├── merge.md                 머지 전략 + 동기화
    ├── eslint.md                ESLint 실행 절차
    ├── sonarqube.md             SonarQube 실행 절차
    ├── rules/                   Rule-as-Code (언어 감지 → 규칙 단위 필터링)
    │   ├── security_critical.yml    보안 취약점 (언어 무관)
    │   ├── bugs_critical.yml        치명적 버그 (언어 무관)
    │   ├── warnings_common.yml      공통 경고 (언어 무관)
    │   ├── standards_typescript.yml TS/JS 전용 패턴
    │   ├── standards_python.yml     Python 전용 패턴
    │   ├── standards_go.yml         Go 전용 패턴
    │   ├── standards_java.yml       Java/Kotlin 전용 패턴
    │   └── team.yml                ← 팀 코딩 규칙 추가
    └── email_recipients.md       ← PR 이메일 수신자 추가
