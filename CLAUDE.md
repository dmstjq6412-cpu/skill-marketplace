## Harness Golden Rule

이 하네스의 존재 이유는 단 하나다: **일관적인 코드 통제와 토큰 사용량의 밸런스를 잡는 것**

Claude는 작업 전 항상 아래 룰을 기준으로 판단한다.

1. **목적 고수** — 모든 결정은 "코드 통제 + 토큰 밸런스"에 기여하는가로 판단한다. 기여하지 않으면 아무리 좋아 보여도 제안하지 않는다.
2. **지금 문제 우선** — 이론적으로 좋아 보이는 것보다 지금 실제로 불편한 것을 먼저 푼다.
3. **단순한 게 이긴다** — 기능이 동작하는 가장 단순한 구조에서 시작한다. 복잡한 설계가 나오면 더 단순한 방법이 없는지 먼저 물어본다. 확장은 실제 불편함이 생겼을 때만 한다 — "더 좋을 것 같아서"는 이유가 아니다.
4. **검증 전엔 확장 없음** — 현재 스킬이 실전에서 잘 동작하는지 확인하기 전에 새 스킬을 추가하지 않는다.
5. **근거 없는 수정 금지** — 측정/데이터 없이 스킬이나 룰을 바꾸지 않는다. "좋아 보여서"는 이유가 아니다. 수정 전 반드시 "무엇이 문제인가"를 수치나 사례로 먼저 확인한다.

> 사용자가 아티클이나 새로운 패턴을 가져오면, 위 룰 기준으로 적용 여부를 먼저 판단하고 오버킬을 경고한다.

## 필수 규칙: git 작업 시 브랜치 확인 (git-guard-claude)

**코드 작업(파일 편집, 기능 구현, 버그 수정)을 시작하기 전, 반드시 현재 브랜치가 올바른지 확인할 것.**

- `main` / `master` / `develop` 브랜치에서 직접 코드를 수정하지 말 것
- 적절한 feature 브랜치가 없으면 git-guard-claude 스킬을 호출해 브랜치를 먼저 생성

> 이유: 잘못된 브랜치에서 작업하면 컨벤션 위반 + 머지 충돌 위험이 생긴다.

## [강제] 코드 작업 전 tdd-guard-claude 실행 — 예외 없음

**⛔ 코드 파일을 작성하거나 수정하기 전에 반드시 `Skill(tdd-guard-claude)`를 실행해야 한다. 이 단계를 건너뛰는 것은 허용되지 않는다.**

적용 대상 (이 중 하나라도 해당하면 무조건 실행):
- 새 함수, 컴포넌트, API 엔드포인트 생성
- 기존 로직 수정 또는 리팩토링
- 버그 수정

예외 — **오직 아래 경우만 면제**:
- CSS/Tailwind 클래스 변경만 있는 경우
- 설정 파일 (vite.config.js, .eslintrc 등)
- package.json 의존성 업데이트

> **위반 시**: 테스트 없이 작성된 코드는 즉시 중단하고, tdd-guard-claude를 실행한 뒤 다시 작업한다.
> "필요 없을 것 같다", "작은 변경이다" 같은 판단으로 건너뛰는 것은 허용되지 않는다.

## [강제] 코딩 제약 규칙 — 예외 없음

### Rule 1: 단일 파일 500줄 제한
> WHY: 500줄 경고는 "지금 당장 멈춰라"가 아니라 "로직을 완성한 뒤 리팩토링하라"는 신호다. 흐름을 끊으면 구현이 불완전한 상태로 남는다.

PostToolUse hook이 500줄 초과 경고를 보내더라도:
- **즉시 작업을 중단하지 않는다**
- 현재 구현 흐름(함수/컴포넌트/핸들러)을 먼저 완성한다
- 완성 후 파일 분리 계획을 사용자에게 보고하고 승인을 받는다

### Rule 2: 테스트 코드 수정 금지
> WHY: 테스트는 의도한 스펙이다. 테스트를 고치면 버그가 고쳐지는 게 아니라 스펙이 바뀌는 것이다.

테스트가 실패했을 때:
- **테스트 코드를 수정하지 않는다**
- 프로덕션 코드를 수정하여 테스트를 통과시킨다
- 테스트 자체가 잘못됐다고 판단되면 사용자에게 먼저 보고하고 승인을 받은 후에만 수정한다

### Rule 3: 오류 발생 시 보고 후 승인
> WHY: 암묵적 수정은 사용자가 모르는 사이에 의도가 바뀐다. 오류는 판단의 재료이지 Claude가 혼자 해결할 문제가 아니다.

테스트 실행 오류, 스킬 수행 오류, **서브에이전트 결과 누락**이 발생하면:
- **암묵적으로 수정하거나 조용히 넘어가지 않는다**
- 오류 내용을 사용자에게 즉시 보고한다
- 수정 방향을 제안하고 승인을 받은 후에만 수정을 진행한다

서브에이전트 결과 누락 시 처리:
- `[Tool result missing due to internal error]` 등 결과를 받지 못한 경우 1회 재시도한다
- 재시도 후에도 실패하면 "서브에이전트 결과를 받지 못했습니다. 계속할까요?" 를 사용자에게 보고하고 대기한다
- 결과 없이 다음 단계로 진행하지 않는다

### Rule 4: 범위 일치 응답
> WHY: "어차피 건드리는 김에"는 의도하지 않은 변경을 만들고 검토 범위를 넓혀 통제를 어렵게 한다.

- 요청과 **인과 관계 없는** 리팩토링, 추상화, 주석 추가는 하지 않는다
- 단, 요청된 변경으로 인해 발생한 회귀/결함은 수정 범위에 포함된다
