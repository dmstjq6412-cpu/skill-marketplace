# 에이전트 프롬프트 템플릿

각 에이전트에 전달하는 표준 템플릿입니다. 플랫폼(Claude/Codex)에 무관하게 이 내용을 채워 사용합니다.

---

## Sequential Code Writer (멀티 모드 전용)

```
아래 파일의 제품 코드를 작성해줘.

- 대상 파일: <파일 경로>
- 레이어: <Layer N — 예: Layer 3 Business Logic>
- 작업 내용: <메인 에이전트가 요약한 변경 사항>
- 이미 완료된 하위 레이어 파일 목록: <Layer 0~N-1 파일 목록>
- 담당 파일만 수정할 것. 다른 파일은 건드리지 말 것
- 다른 에이전트가 만든 변경을 되돌리지 말 것
```

## Unit Test Worker

```
아래 파일에 대한 유닛/컴포넌트 테스트를 작성해줘.

- 대상 파일: <파일 경로>
- 파일 내용: <변경된 파일 전체 내용>
- 예상 테스트 파일: <메인 에이전트가 계산한 경로>
- 테스트 프레임워크: <감지 결과 — 예: vitest>
- Mock 방식: <감지 결과 — 예: vi.mock / vi.fn()>
- Mock 초기화: <감지 결과 — 예: vi.clearAllMocks()>
- 기존 테스트 파일이 있으면 덮어쓰지 말고 케이스를 추가할 것
- DB 및 외부 의존성은 위에서 명시한 mock 방식으로 처리
- 80% 커버리지 목표: happy path + 오류 처리 + 경계값 포함
- 테스트 실행하지 말 것 (실행은 git-convention PreCommit이 담당)
- 이 테스트 파일만 담당한다. 다른 파일은 수정하지 말 것
- 다른 에이전트가 만든 변경을 되돌리지 말 것
```

## Impact Analyzer

```
아래 변경사항이 기존 어떤 흐름에 영향을 주는지 분석해줘.

변경된 파일:
<파일 목록>

git diff:
<diff 내용>

반환 형식 (JSON만 반환, 설명 텍스트 없이):
{
  "affected_flows": [
    {
      "name": "흐름 이름",
      "files_involved": ["파일1", "파일2"],
      "description": "어떤 동작이 영향받는지 한 줄 설명"
    }
  ]
}

영향받는 흐름이 없으면 { "affected_flows": [] } 를 반환할 것.
이 작업은 읽기 전용이다. 파일은 수정하지 말 것.
```

> **Analyzer 결과 처리:** `affected_flows`가 비어있거나 JSON 파싱이 실패하면 Phase 2를 건너뜁니다. 파싱 실패 시 메인 에이전트가 Analyzer 출력을 직접 읽고 회귀 여부를 판단합니다.

## Regression Writer

```
아래 흐름들에 대해 회귀 테스트를 작성해줘.
기존 동작이 변경 후에도 깨지지 않는지 검증하는 게 목적이야.

영향받는 흐름:
<Analyzer 결과 JSON>

- 각 흐름의 기존 핵심 시나리오를 테스트로 작성
- 변경으로 인해 깨질 수 있는 엣지 케이스 포함
- 테스트 실행하지 말 것 (실행은 git-convention PreCommit이 담당)
- 담당 테스트 파일 범위를 명확히 지킬 것
- 다른 에이전트가 만든 변경을 되돌리지 말 것
```
